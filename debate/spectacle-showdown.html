Act IV

Cross-Examination

5 Against Claude

1. The test suite is theater +

30 test functions exist. Zero pass. The tests import from a module path that was refactored but never updated. This is a test suite that has never run green against the current code. In a security tool, untested code is untrustworthy code.

2. All secrets look identical +

Every Anthropic API key becomes sk-ant-***REDACTED***. If your transcript contained three different leaked keys, the scrubbed output gives you no way to tell them apart. For forensic review, this is a critical blind spot.

3. Pattern overlap is a land mine +

The OpenAI pattern matches before the Anthropic pattern can fire. No overlap resolution engine exists. The same string can be partially redacted by one pattern, then mangled by another -- the classic "regex chain of destruction."

4. HTML scrubbing is naive +

The HTML parser splits on <[^>]+> and scrubs both tags and text nodes. Any attribute containing a > character, any inline JavaScript, any CDATA section will break the split and potentially leak secrets.

5. No data model, no contract +

There is no intermediate representation. Raw strings in, scrubbed strings out. The scrubber cannot tell you where in the transcript structure a secret was found. Debugging false positives becomes a manual archaeology exercise.

5 Against Team

1. HTML is a foreign language +

The team branch cannot read HTML at all. Its parser registry contains exactly two parsers: JSONL and JSON Web. If someone hands it an HTML transcript export, it throws a ValueError. A format gap real users will hit.

2. Twelve missing provider patterns +

Vercel tokens, Supabase keys, Cloudflare API tokens, Netlify tokens, SendGrid keys, Twilio keys, npm tokens, PyPI tokens, Google API keys -- all detected by claude, all invisible to team. Those missing patterns are real secrets walking through undetected.

3. No CI exit codes +

The CLI always exits 0 whether it found 100 secrets or none. In a CI/CD pipeline, the scan command cannot gate a deployment. For the DevSecOps use case, this is a showstopper.

4. Four dependencies, not one +

click, rich, tomli, tomli-w. The Rich dependency alone pulls in a tree of sub-dependencies. For a security-sensitive tool, a minimal dependency surface is a feature. Every dependency is an attack surface.

5. 3x code for the same job +

2,953 lines to do what 966 lines also do. Three times the surface area for bugs, three times the code to audit, three times the maintenance burden. The 4-layer architecture is elegant -- but is it necessary for a regex-and-replace tool?

The Verdict

Neither branch is ready for production.

But if forced to choose -- and we are forced to choose...

The Team Branch Wins.

A tested, architecturally sound codebase with missing features is a solvable problem. Taking an untested monolith and retrofitting it with stable numbering, overlap resolution, and round-trip fidelity? That is a rewrite.
Epilogue

When to Use What

Quick one-off scrub
Choose claude
Fix the import path, verify the patterns, and run it. Single-file simplicity -- read the entire codebase in 15 minutes.
Team workflow / CI pipeline
Choose team
Architecture built for this. Swap components without rewriting the pipeline. Stable numbered redactions for code review clarity.
Security compliance audit
Choose team
Overlap resolution and stable numbering are requirements. Auditable trail from original to replacement.
Maximum detection breadth
Merge both
Claude's 39 patterns in Team's scanner registry. The best catalog meets the most robust engine.

Speed without correctness is dangerous. Correctness without coverage is incomplete. The best tool ships both -- and neither branch has arrived there yet.

But the path from team is shorter, straighter, and paved with 257 green tests.
